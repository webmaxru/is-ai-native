# Migration Plan: Frontend to Azure Static Web Apps

**Status:** Proposed
**Date:** 2025-07-22
**Author:** Maxim Salnikov (with Copilot assistance)

---

## 1. Problem Statement

The "Is it AI-Native?" web application currently runs as a single Express container on Azure Container Apps, serving both the vanilla JS SPA frontend and the REST API from the same process on port 3000.

This works but is suboptimal:

- **No CDN edge caching** — static assets (JS, CSS, images) are served from a single Azure region.
- **Wasted container I/O** — the container handles static file requests that don't need server-side logic.
- **Coupled scaling** — scaling the API also scales the static file server, and vice versa.
- **No scale-to-zero benefit** — to keep the frontend responsive, the container must stay warm even when there's no API traffic.

## 2. Target Architecture

```
                     ┌─────────────────────────────────┐
                     │     Azure Static Web Apps        │
                     │   (Free tier, global CDN, SSL)   │
                     │                                  │
  Browser ─────────► │  Static: *.js, *.css, images     │
                     │                                  │
                     │  Everything else ─────────────────────► Azure Container Apps
                     │   (linked backend proxy)          │   (Express: API + SSR)
                     │                                  │    scale-to-zero
                     └─────────────────────────────────┘
                               ▲
                               │
                     isainative.dev (custom domain)
```

### What SWA serves directly (from CDN)

- `webapp/frontend/src/*.js` — application JavaScript
- `webapp/frontend/src/*.css` — stylesheets
- `webapp/frontend/assets/**` — images, fonts, brand assets
- Favicon files placed at root-level paths (`/favicon.ico`, `/favicon.svg`, etc.)

### What SWA proxies to Container App (linked backend)

- `/api/*` — REST API endpoints (`/api/scan`, `/api/config`, `/api/report/:id`, `/api/health`)
- HTML page requests (`/`, `/{owner}/{repo}`, `/_/report/:id`) — server-side rendered by Express with per-page SEO metadata (title, OG tags, structured data)
- `/robots.txt`, `/sitemap.xml`, `/site.webmanifest` — dynamically generated by Express
- `/manifest.webmanifest` — redirect to `/site.webmanifest` (handled by Express)

### Key architectural decisions

| Decision | Rationale |
|----------|-----------|
| **No SWA Azure Functions** | SSR stays in Express. SWA managed functions and linked backends conflict on `/api/*` — you cannot use both simultaneously. |
| **No CORS configuration** | SWA linked backend makes API calls same-origin. No cross-origin requests. |
| **No code duplication** | All server-side logic (API, SSR, dynamic files) stays in the Express container. |
| **Container App: scale-to-zero** | Lowest cost. Container only runs when there's actual API/SSR traffic. |
| **Cold-start UX** | Frontend shows a loader/spinner explaining "app is starting" while the Container App wakes up. |
| **SWA Free tier** | Frontend is ~200 KB (vanilla JS, no framework). Free tier limits (0.5 GB app size, 100 GB bandwidth, 2 custom domains) are more than sufficient. |
| **Custom domain on SWA** | `isainative.dev` points to SWA for global CDN and free managed SSL. |
| **All-at-once deployment** | Not phased — ship the full migration in a single coordinated release. |

## 3. Current Coupling Analysis

| Coupling point | Current behavior | Migration impact |
|----------------|-----------------|-----------------|
| Static file serving | Express serves `webapp/frontend/` via `express.static()` (`server.js:374-381`) | SWA serves from CDN; Express static middleware becomes unused for public traffic |
| HTML template rendering | Express replaces `__PAGE_TITLE__`, `__PAGE_DESCRIPTION__`, `__PAGE_CANONICAL__`, OG tags via `site-metadata.js` (`server.js:382-388`) | **Stays in Express** — SWA routes HTML requests to linked backend |
| API calls | Frontend uses relative `/api/` paths (`api.js:1`: `const API_BASE = '/api'`) | Works as-is — linked backend keeps same-origin |
| robots.txt / sitemap.xml | Express generates dynamically (`server.js:351-357`) | **Stays in Express** — accessed via linked backend |
| Web manifest | Express generates `site.webmanifest` + redirect from `manifest.webmanifest` (`server.js:359-365`) | **Stays in Express** — accessed via linked backend |
| Favicon aliases | Express maps 13 alias paths to `assets/brand/` files (`server.js:26-38`) | SWA serves these as static files at root-level paths; Express aliases become redundant |
| CSP headers | Helmet middleware on Express (`server.js:329-335`) | Stays for dynamic routes; SWA adds its own headers for static assets |
| Dockerfile | Lines 33-36 copy frontend into container | Remove frontend copy; container ships backend + core only |
| CD workflow | Single deploy: build image → push GHCR → deploy Container App (`cd.yml:113-162`) | Two deploys: SWA (static assets) + Container App (API + SSR) |
| Custom domains | Bound to Container App ingress (`cd.yml:158-161`) | Move to SWA; Container App only receives traffic via linked backend |
| WebMCP | Client-side browser API in `webmcp.js` — no server dependency | No impact — works on any origin |
| Application Insights | Connection string passed to frontend via `/api/config` | No impact — still fetched via API |

## 4. SEO Strategy

**No changes needed.** Server-side HTML rendering stays in Express on the Container App.

- SWA routes all non-static requests (including `/`, `/{owner}/{repo}`, `/_/report/:id`) to the linked backend.
- Express continues to inject per-page SEO metadata exactly as today.
- Social media crawlers hit: SWA → linked backend → Express → rendered HTML with correct meta tags.
- `robots.txt` and `sitemap.xml` are dynamically generated by Express and served via linked backend.

## 5. Cold-Start UX (New Requirement)

With scale-to-zero on the Container App, the first request after an idle period may take 10-20 seconds while the container starts. Since SWA serves static assets instantly from CDN, the page shell loads fast — but API calls and SSR will hang until the container is ready.

### Design

1. **For SSR pages** (HTML requests proxied to linked backend):
   - SWA's linked backend returns a gateway timeout or hangs during cold start.
   - The frontend `index.html` (served by SWA as static) should include an inline cold-start detection mechanism.
   - When the initial `/api/config` call fails or times out, show a branded spinner/loader with the message: **"App is starting up, please wait…"**
   - Retry the API call with exponential backoff (e.g., 2s, 4s, 8s).
   - Once the API responds, hide the spinner and render normally.

2. **Implementation approach:**
   - Add a lightweight inline `<script>` in `index.html` that shows a CSS-only spinner overlay.
   - The spinner displays immediately and is hidden once `app.js` successfully completes initialization.
   - If `api.js`'s `fetchConfig()` fails, the spinner message updates to "App is starting up, please wait…"
   - No external dependencies needed — pure CSS animation + vanilla JS.

3. **For direct API calls** (`/api/scan`, `/api/report/:id`):
   - These are triggered by user actions (clicking "Scan"), not on page load.
   - The existing loading states in the UI already cover this case.
   - The `/api/health` endpoint can be used as a warm-up probe.

### UX flow

```
[Page loads from SWA CDN — instant]
    │
    ├─ Static assets load (JS, CSS, images) — served by CDN
    │
    └─ app.js calls /api/config
         │
         ├─ Success → app renders normally (Container App was warm)
         │
         └─ Timeout/error → Show spinner: "App is starting up, please wait…"
              │
              └─ Retry with backoff → Success → hide spinner, render app
```

## 6. Implementation Tasks

> All tasks ship together in a single coordinated PR/release.

### Task 1 — Prepare Static Assets for SWA (S)

**Scope:** `webapp/frontend/`

- Verify `webapp/frontend/` directory is deployable as pure static files.
- Place favicon and brand asset files at root-level SWA paths so they're served without Express aliases:
  - Copy/symlink `assets/brand/favicon.ico` → `favicon.ico` (at SWA root)
  - Copy/symlink `assets/brand/favicon.svg` → `favicon.svg`
  - Same for `apple-touch-icon.png`, `android-chrome-*.png`, `maskable-icon-*.png`, `mask-icon.svg`, `mstile-*.png`, `social-card.png`, `social-card.svg`
- Confirm all frontend JS uses relative `/api/` paths (already true: `api.js` line 1).
- Define the SWA deployment artifact path: `webapp/frontend/`.

### Task 2 — Create SWA Configuration (S)

**Scope:** `webapp/frontend/staticwebapp.config.json`

Create `staticwebapp.config.json` with:

```jsonc
{
  "routes": [
    // Static assets: let SWA serve them directly (implicit, no rule needed)

    // API routes: proxy to linked backend
    { "route": "/api/*", "allowedRoles": ["anonymous"] },

    // Dynamic files: proxy to linked backend
    { "route": "/robots.txt", "allowedRoles": ["anonymous"] },
    { "route": "/sitemap.xml", "allowedRoles": ["anonymous"] },
    { "route": "/site.webmanifest", "allowedRoles": ["anonymous"] },
    { "route": "/manifest.webmanifest", "allowedRoles": ["anonymous"] }
  ],
  "responseOverrides": {
    "404": {
      "rewrite": "/.linked_backend/"
    }
  },
  "globalHeaders": {
    "Cache-Control": "public, max-age=3600"
  },
  "platform": {
    "apiRuntime": "node:20"
  }
}
```

Key routing logic:
- SWA attempts to match a request to a static file first.
- If no static file matches, the `404` response override rewrites to the linked backend.
- This means all HTML page requests (`/`, `/{owner}/{repo}`, `/_/report/:id}`) automatically go to Express for SSR.
- `/api/*`, `/robots.txt`, `/sitemap.xml`, `/site.webmanifest` are explicitly routed (they also fall through to linked backend).
- `navigationFallback` is **not used** — Express handles SPA routing via the linked backend.

### Task 3 — Implement Cold-Start Spinner UX (S)

**Scope:** `webapp/frontend/index.html`, `webapp/frontend/src/app.js`, `webapp/frontend/src/main.css`

- Add an inline CSS-only spinner overlay in `index.html` (visible by default, hidden by JS on successful init).
- Update `app.js` initialization to:
  - Catch `fetchConfig()` failures/timeouts.
  - Show "App is starting up, please wait…" message in the spinner.
  - Retry with exponential backoff (2s → 4s → 8s, max 3 retries).
  - Hide spinner on success.
- Ensure the spinner works even before any external JS/CSS loads (inline styles and script).

### Task 4 — Add SWA Bicep Infrastructure (M)

**Scope:** `infra/main.bicep`

Add new resources:

```bicep
// Azure Static Web Apps resource
resource staticWebApp 'Microsoft.Web/staticSites@2023-12-01' = {
  name: '${namePrefix}-swa'
  location: location  // SWA Free tier: limited regions (e.g., centralus, eastus2, westus2)
  tags: tags
  sku: {
    name: 'Free'
    tier: 'Free'
  }
  properties: {
    buildProperties: {
      skipGithubActionWorkflowGeneration: true  // We manage our own CD workflow
    }
  }
}

// Linked backend: SWA proxies dynamic requests to Container App
resource linkedBackend 'Microsoft.Web/staticSites/linkedBackends@2023-12-01' = {
  parent: staticWebApp
  name: 'container-app-backend'
  properties: {
    backendResourceId: containerApp.id
    region: location
  }
}
```

Additional Bicep changes:
- Add `swaDeploymentToken` output (needed by CI/CD to deploy static assets).
- Add `swaUrl` output for smoke tests.
- Migrate custom domain (`isainative.dev`) from Container App to SWA:
  - Add `Microsoft.Web/staticSites/customDomains` resource.
  - Remove custom domain binding from Container App ingress.
- Switch `containerStartupStrategy` parameter default and CD override from `keep-warm` to `scale-to-zero`.
- Container App ingress should remain enabled (linked backend needs it) but custom domain moves to SWA.

### Task 5 — Update CI/CD Pipelines (M)

**Scope:** `.github/workflows/cd.yml`, `.github/workflows/ci.yml`

**CD workflow changes (`cd.yml`):**
- Add SWA deployment step after Bicep deployment:
  ```yaml
  - name: Deploy frontend to SWA
    uses: Azure/static-web-apps-deploy@v1
    with:
      azure_static_web_apps_api_token: ${{ steps.bicep.outputs.swaDeploymentToken }}
      action: 'upload'
      app_location: 'webapp/frontend'
      skip_app_build: true  # No build step — vanilla JS
      skip_api_build: true  # No SWA functions
  ```
- Change `containerStartupStrategy` parameter from `keep-warm` to `scale-to-zero` (line 154 of current `cd.yml`).
- Update smoke tests to hit the SWA URL instead of the Container App URL.
- Add SWA-specific smoke test: verify static assets return 200 from CDN.
- Update the "Print application URL" step to show SWA URL.

**CI workflow changes (`ci.yml`):**
- Add validation step: lint/validate `staticwebapp.config.json` (JSON syntax check).
- Optionally: SWA CLI preview deployment for PRs.

### Task 6 — Slim Down the Container (S)

**Scope:** `Dockerfile`, `webapp/backend/src/server.js`

**Dockerfile changes:**
- Remove lines 33-36 (frontend asset copy):
  ```dockerfile
  # REMOVE these lines:
  COPY webapp/frontend/index.html ../frontend/index.html
  COPY webapp/frontend/assets ../frontend/assets
  COPY webapp/frontend/src ../frontend/src
  ```
- Container now ships backend + core package only.

**Server changes:**
- The `express.static()` middleware and favicon alias routes can remain in `server.js` for now — they'll simply never receive traffic because SWA serves those paths.
- Optionally: guard them behind a `FRONTEND_PATH` check (already partially done — `server.js:373`: `if (runtime.frontendPath && runtime.frontendTemplate)`).
- The HTML catch-all route (`server.js:382-388`) **must stay** — linked backend still routes HTML requests to Express for SSR.
- In production, `resolveFrontendPath()` will return `null` (no frontend directory in container), so `runtime.frontendTemplate` will be `null`, and the SSR catch-all won't activate.

**⚠️ CRITICAL:** The SSR catch-all route (`server.js:382`) is gated behind `if (runtime.frontendPath && runtime.frontendTemplate)`. Once frontend files are removed from the container, this block won't execute — meaning HTML page requests from the linked backend will hit the `notFoundHandler` instead of rendering SEO metadata.

**Fix required:** The SSR route must be decoupled from `runtime.frontendPath`. Options:
1. **Load the HTML template from a different source** — embed `index.html` as a string constant, or load it from a config/environment variable.
2. **Ship `index.html` alone in the container** (keep just the template, not all static assets) — smallest change.
3. **Fetch `index.html` from SWA at startup** — adds a runtime dependency on SWA.

**Recommended approach:** Option 2 — keep only `index.html` in the container (single file, ~5 KB) for SSR template use. Update Dockerfile:
```dockerfile
# Keep ONLY the HTML template for SSR (not the full frontend)
COPY webapp/frontend/index.html ../frontend/index.html
```
Remove the other two COPY lines (`assets` and `src`). This way `resolveFrontendPath()` still finds the directory, `loadFrontendTemplate()` loads the template, and the SSR catch-all activates — but SWA handles all actual static asset serving.

### Task 7 — Custom Domain Migration (M)

**Scope:** DNS, `infra/main.bicep`, GitHub Actions secrets

This is a manual/semi-automated step with a brief DNS propagation window.

1. **Before cutover:**
   - Lower DNS TTL for `isainative.dev` to 60 seconds (at least 24h before cutover).
   - Verify SWA is fully working on its default `*.azurestaticapps.net` URL.
   - Test all routes: static assets, API proxying, SSR pages, dynamic files.

2. **DNS cutover:**
   - Add custom domain to SWA resource (Bicep or CLI: `az staticwebapp hostname set`).
   - Update DNS records to point `isainative.dev` to SWA's default hostname (CNAME).
   - SWA provisions a free managed SSL certificate automatically.

3. **After cutover:**
   - Wait for DNS propagation (minutes to hours depending on upstream caches).
   - Verify HTTPS works on `isainative.dev`.
   - Remove old custom domain binding from Container App Bicep config.
   - Update GitHub Actions secrets (`CUSTOM_DOMAIN_NAME`, `MANAGED_CERT_NAME`) or remove them if no longer needed.
   - Restore DNS TTL to normal (3600s).
   - Update `SITE_ORIGIN` secret to keep using `https://isainative.dev`.

4. **Rollback plan:**
   - If issues arise, revert DNS to point back to Container App.
   - Re-add the frontend to the container (revert Dockerfile changes).
   - Container App still has the full application — zero data loss.

### Task 8 — Update Monitoring and Documentation (S)

## 7. Risks and Mitigations

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| **SSR route breaks after removing frontend from container** | HTML pages return 404 instead of rendered SEO content | High (if not addressed) | Keep `index.html` in container for SSR template (Task 6 recommendation) |
| **Container App cold start via linked backend** | First request after idle takes 10-20s | Medium | Cold-start spinner UX (Task 3); `/api/health` as warm-up probe |
| **Custom domain DNS propagation** | Brief period where some users hit old endpoint | Low | Lower TTL 24h before cutover; test on staging domain first |
| **SWA routing misconfiguration** | Wrong requests get SWA 404 instead of reaching linked backend | Medium | Test all route patterns before DNS cutover; use SWA preview URL |
| **Static asset cache staleness** | Users see old CSS/JS after deploy | Low | Set conservative `max-age` (3600s); consider adding content hashing later |
| **SWA Free tier limits** | Hit 100 GB bandwidth or 0.5 GB storage | Very Low | Frontend is ~200 KB; bandwidth would need millions of visits/month |
| **SEO regression during cutover** | Social preview cards briefly incorrect | Low | Test with social media debuggers before DNS cutover; Container App remains as fallback |
| **Linked backend latency** | Additional network hop adds latency to API calls | Very Low | SWA and Container App in same region; linked backend adds <5ms |

## 9. What Does NOT Change

- **Core scan engine** (`packages/core/`) — untouched.
- **API routes and logic** — all stay in Express on Container App.
- **SSR / SEO metadata** — Express continues to render HTML with per-page metadata.
- **VS Code extension, GitHub CLI extension, standalone CLI** — unaffected (they don't use the web app infrastructure).
- **Application Insights telemetry** — connection string still served via `/api/config`.
- **WebMCP integration** — client-side only, works on any origin.
- **Report sharing** — Azure Files mount stays on Container App.
- **Rate limiting** — stays in Express middleware.

## 10. Rollback Plan

If the migration causes issues after going live:

1. **Immediate (< 5 min):** Revert DNS for `isainative.dev` to point back to Container App.
2. **Short-term:** Re-add frontend files to Dockerfile and redeploy container.
3. **The Container App always has the full backend** — SSR, API, and dynamic files continue to work.
4. **No data loss** — reports, configuration, and shared reports are stored in Azure Files.

The rollback is safe because the Container App retains all server-side logic throughout the migration.
